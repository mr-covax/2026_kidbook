# Межпроцессное взаимодействие (IPC)

## Определение

**Межпроцессное взаимодействие** (IPC, от английского Inter-Process Communication) — это механизмы, которые позволяют [процессам](process.md) обмениваться данными и согласовывать свои действия. 

**Процесс** — это запущенная программа со своей памятью и ресурсами. Процессы изолированы друг от друга: один процесс не может напрямую прочитать память другого. Это нужно для стабильности, но создаёт проблему: как процессам работать вместе?

IPC решает эту проблему, предоставляя безопасные каналы для обмена информацией между процессами.

## Подробное описание

### Почему процессы не видят друг друга

Каждый процесс работает в своей собственной области памяти. Это как если бы каждый ребёнок играл в своей комнате с закрытой дверью. Так компьютер защищает процессы: если один процесс перестанет работать, другие не пострадают.

Но иногда процессам нужно сотрудничать. Например, один процесс загружает картинку из интернета, а другой показывает её на [экране](window_manager.md). Для такого сотрудничества нужны специальные механизмы — IPC.

```mermaid
flowchart LR
    subgraph Компьютер["Компьютер"]
        P1[Процесс 1]
        P2[Процесс 2]
        P3[Процесс 3]
    end
    
    P1 -.->|Изолирован| P2
    P2 -.->|Изолирован| P3
    P1 ~~~|IPC| P3
```

### Трубы (Pipes)

**Труба** — это канал для передачи данных в одном направлении. Один процесс записывает данные в трубу, другой процесс читает их. Данные в трубе можно прочитать только один раз.

Труба похожа на настоящий водопровод: вода течёт в одну сторону, и если её не забрать сразу, она уйдёт дальше.

```mermaid
flowchart LR
    A[Процесс-отправитель] -->|Записывает байты| B[Труба]
    B -->|Читает байты| C[Процесс-получатель]
    style B fill:#e1f5fe
```

**Особенности труб:**
- Данные текут только в одну сторону
- Данные читаются по порядку, как в очереди
- После прочтения данные исчезают из трубы

### Сообщения (Messages)

**Сообщения** — это способ обмена отдельными порциями данных. В отличие от труб, сообщения имеют границы: каждое сообщение — это законченная единица информации.

Процесс отправляет сообщение в очередь, другой процесс забирает его. Сообщения могут иметь разный размер и тип.

```mermaid
flowchart TB
    A[Процесс А] -->|Отправляет| B[Очередь сообщений]
    C[Процесс Б] -->|Отправляет| B
    B -->|Получает| D[Процесс В]
    B -->|Получает| E[Процесс Г]
    style B fill:#fff9c4
```

**Особенности сообщений:**
- Каждое сообщение отдельно адресуется
- Получатель выбирает, какое сообщение читать
- Подходит для команд и структурированных данных

### Общая память (Shared Memory)

**Общая память** — это область памяти, которую несколько процессов могут читать и записывать одновременно. Это самый быстрый способ обмена данными, потому что процессы обращаются к данным напрямую, без копирования.

```mermaid
flowchart TB
    subgraph Shared["Общая память"]
        M[Данные: 12345]
    end
    
    P1[Процесс 1] <-->|Чтение/запись| M
    P2[Процесс 2] <-->|Чтение/запись| M
    style Shared fill:#e8f5e9,stroke:#2e7d32
```

**Проблема общей памяти:** если два процесса одновременно изменят одни и те же данные, результат будет непредсказуем. Поэтому общую память используют вместе с механизмами синхронизации.

### Семафоры (Semaphores)

**Семафор** — это счётчик, который помогает процессам договариваться о доступе к общему ресурсу. Семафор показывает, сколько процессов могут одновременно использовать ресурс.

Пример: семафор со значением 3 разрешает трём процессам одновременно войти в критическую секцию. Четвёртый процесс будет ждать.

```mermaid
flowchart TD
    S[Семафор: значение = 2]
    
    S -->|Процесс 1: занял| S1[Семафор: 1]
    S1 -->|Процесс 2: занял| S2[Семафор: 0]
    S2 -->|Процесс 3: ждёт| W[Ожидание]
    S1 -->|Процесс 1: освободил| S
    style S fill:#ffe0b2
    style S2 fill:#ffcdd2
```

**Семафоры решают задачу координации:** они не дают превысить лимит одновременного доступа к ресурсу.

### Спинлоки (Spinlocks)

**Спинлок** — это простая блокировка, при которой процесс постоянно проверяет, свободен ли ресурс, вместо того чтобы перейти в режим ожидания.

Спинлок похож на человека, который стоит у двери и каждые две секунды дёргает ручку, проверяя, не открылась ли она.

```mermaid
flowchart LR
    A[Нужен ресурс] --> B{Свободен?}
    B -->|Да| C[Занять ресурс]
    B -->|Нет| D[Проверить снова]
    D --> B
    C --> E[Освободить]
    style D fill:#fff3e0
```

**Когда использовать спинлоки:** только для очень коротких ожиданий. Если ресурс занят надолго, процесс тратит время на бесполезные проверки.

### RW-локи (Read-Write Locks)

**RW-лок** (блокировка чтения-записи) различает два типа доступа:
- **Чтение** — много процессов могут читать данные одновременно
- **Запись** — только один процесс может изменять данные, и в этот момент никто не читает

Это эффективно, когда данные часто читают, но редко меняют.

```mermaid
flowchart TB
    subgraph RW["RW-лок"]
        direction TB
        R1[Читатель 1] & R2[Читатель 2] & R3[Читатель 3]
        W[Писатель]
    end
    
    R1 & R2 & R3 -.->|Совместный доступ| R1
    W -.->|Эксклюзивный доступ| W
    style R1 fill:#e3f2fd
    style W fill:#fce4ec
```

### Прерывания в режиме пользователя (Usermode Interrupts)

**Прерывания в режиме пользователя** — это сигнал, который один процесс может отправить другому, чтобы привлечь внимание. Получив сигнал, процесс может временно прервать свою работу и обработать событие.

Пример: процесс-таймер отправляет сигнал процессу-будильнику, когда наступает нужное время.

```mermaid
flowchart LR
    A[Процесс-источник] -->|Сигнал| B[Процесс-получатель]
    B -->|Прерывает текущую задачу| C[Обработчик сигнала]
    C -->|Возвращается| B
    style C fill:#f3e5f5
```

**Зачем нужны прерывания:** позволяют реагировать на срочные события без постоянного опроса.

### Сравнение механизмов IPC

| Механизм | Скорость | Направление данных | Основная цель |
|----------|----------|-------------------|---------------|
| **Трубы** | Средняя | Одно направление | Передача [потока](thread.md) байтов |
| **Сообщения** | Средняя | Любое | Обмен законченными порциями данных |
| **Общая память** | Очень высокая | Любое | Быстрый доступ к общим данным |
| **Семафоры** | Высокая | — | Координация доступа к ресурсу |
| **Спинлоки** | Очень высокая | — | Кратковременная защита данных |
| **RW-локи** | Высокая | — | Разделение доступа на чтение и запись |
| **Прерывания** | Мгновенная | Сигнал | Срочное уведомление процесса |

### Краткое резюме

Межпроцессное взаимодействие существует, потому что процессы изолированы, но должны сотрудничать.

**Для передачи данных:**
- Трубы — простой поток в одну сторону
- Сообщения — отдельные порции с границами
- Общая память — прямой и быстрый доступ

**Для координации:**
- Семафоры — счётчик доступных мест
- Спинлоки — активное ожидание блокировки
- RW-локи — разделение читателей и писателей

**Для уведомлений:**
- Прерывания — мгновенный сигнал процессу

Выбор механизма зависит от задачи: что передаётся, как часто, и нужна ли синхронизация.
